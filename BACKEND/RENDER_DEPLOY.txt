TRIPLINK backend on Render (short checklist)
==========================================

1) Push this repo to GitHub (or GitLab) if it is not there yet.

2) In Render: New → Blueprint → connect the repo → pick render.yaml at the repo root.
   OR: New → Web Service → connect repo, set Root Directory = BACKEND, same Build/Start as in render.yaml.

3) Create PostgreSQL: New → PostgreSQL (free tier if available). Copy the Internal/External DATABASE_URL.

4) On the web service → Environment → add:
   DATABASE_URL = <paste from Postgres, usually starts with postgres://>
   (Optional) RENDER_EXTERNAL_URL = https://YOUR-SERVICE.onrender.com  if hosts/CSRF need a hint.

5) Deploy. Logs should show migrate + daphne listening.

6) Test: open https://YOUR-SERVICE.onrender.com/admin/ in a browser.

7) Mobile app: set FRONTEND/TRIPLINK/.env
   EXPO_PUBLIC_API_BASE=https://YOUR-SERVICE.onrender.com
   Rebuild the APK (gradle assembleRelease).

Notes
-----
- Free web services sleep after idle; first request can be slow.

Persistent uploads (survive redeploy)
-------------------------------------
Without extra config, the app stores files on the instance disk (ephemeral on Render). Use one of:

A) S3 or Cloudflare R2 (recommended; works on free tier)
   In the web service → Environment, set:
   - AWS_STORAGE_BUCKET_NAME
   - AWS_ACCESS_KEY_ID
   - AWS_SECRET_ACCESS_KEY
   - AWS_S3_REGION_NAME   (e.g. auto for R2, or us-east-1 for AWS)
   Optional for R2:
   - AWS_S3_ENDPOINT_URL  (https://<accountid>.r2.cloudflarestorage.com)
   - AWS_S3_CUSTOM_DOMAIN (your public bucket hostname, e.g. pub-xxx.r2.dev or your domain)
   Optional prefix inside the bucket:
   - AWS_S3_LOCATION      (e.g. media)
   Make the bucket (or prefix) publicly readable for GET so image URLs load in the app; ACLs are not used
   (AWS_DEFAULT_ACL=None). After enabling, new uploads go to object storage; old rows still point at
   old /media/ paths until you re-upload or migrate files.

B) Render Persistent Disk (paid web services)
   Add a disk in the Render dashboard (or blueprint disk: mountPath + sizeGB), mount e.g. at
   /var/triplink-media, then set environment variable MEDIA_ROOT=/var/triplink-media (no S3 vars).
   The API will keep serving /media/ from that path across deploys.

- WebSockets use in-memory channel layer on one dyno; scale-out would need Redis + channels_redis.
